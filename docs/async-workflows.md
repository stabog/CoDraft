# Async-подрежимы

Внутри `collaborationMode: async` — два подрежима. Выбираются при создании документа.

## Приоритет реализации

| Подрежим | Статус |
|----------|--------|
| **Owner hub** | MVP — реализуем первым ([ADR-009](./decisions.md#adr-009-owner-hub--первый-реализуемый-async-подрежим)) |
| **Handoff** | После owner hub; общая база версий и комментариев переиспользуется |

До появления выбора при создании все новые документы в прототипе — `asyncWorkflow: owner_hub`.

---

## Handoff (по очереди)

> **Статус:** описан, в коде — фаза 2.


**Метафора:** пересылка Word — «мяч у тебя».

### Поток

1. Создатель → **v1**, `currentActor` = создатель (или сразу передача).
2. Actor правит draft, комментирует версии, нажимает **«Зафиксировать и передать»** → новая версия + `handoff.to` + смена `currentActor`.
3. Следующий участник видит diff, комментирует, правит, фиксирует и передаёт обратно.

### Когда выбирать

- Два равных соавтора, быстрый пинг-понг.
- Юрист/редактор как **соавтор**, а не советник.
- Нужна дисциплина «один редактор в момент времени».

### Крайние случаи

- Получатель не отвечает → создатель может **забрать ход** (reclaim).
- Один получатель на handoff; цепочка B → C → A — последовательная передача.

## Owner hub (владелец + предложения)

> **Статус:** MVP — основной подрежим прототипа.

**Метафора:** RFC / maintainer — «вот моя версия, присылайте правки, я решу».

### Поток

1. Owner создаёт документ → **v1**.
2. Участники **параллельно**: комментарии к v1 + **proposals** (полный предложенный текст).
3. Owner видит входящие, diff `v1 ↔ каждый proposal`, собирает draft.
4. Owner фиксирует **v2** (опционально `incorporatedProposalIds`). Proposals к v1 → `superseded`.

### Когда выбирать

- Один ответственный + много ревьюеров.
- Product brief, политика, договор (owner подписывает итог).
- Несколько LLM-вариантов с одной базы.
- Параллельная обратная связь без очереди.

### Слияние в одну ветку

Паттерн «open branches» (несколько v2 от одного parent) **сворачивается** в owner hub:

| Open (DAG) | Owner hub |
|------------|-----------|
| v2a, v2b — sibling versions | proposal A, proposal B — вне main |
| merge-узел с двумя parents | owner фиксирует v2 с одним parent |

Коллективность сохраняется; история для читателя остаётся линейной.

Чистый Open (равные права на canonical без арбитра) — **не в MVP**, только если появится явный запрос.

## Сравнение подрежимов

| | Handoff | Owner hub |
|---|---------|-----------|
| Кто правит draft | По очереди (`currentActor`) | Только owner |
| Параллельные правки | Нет (норма) | Да (proposals) |
| Фиксация версии | Actor + передача хода | Owner |
| Слияние | Не требуется | Owner вручную в draft |
| LLM | Один агент по очереди | Несколько proposals |
| Сложность UI | Низкая | Средняя (панель входящих) |

## Примеры

### Договор: вы + юрист

- **Owner hub**, если вы единственный ответственный за текст; юрист шлёт proposal.
- **Handoff**, если юрист — полноценный соавтор по очереди.

### Бриф: вы + 5 коллег

- **Owner hub** — все комментируют и шлют proposals параллельно, вы собираете v2.

### Пост: два маркетолога

- **Handoff** — быстрый пинг-понг без лишнего слоя «предложений».

### Текст + несколько LLM

- **Owner hub** — варианты как proposals, сравнение diff с v1.

### Политика: HR + юрист + security

- **Owner hub** — HR owner, остальные proposals + комментарии, финальная v3 официальная.

## UX (ориентиры)

**Handoff:** badge «Твой ход», кнопка «Зафиксировать и передать» (кому + summary).

**Owner hub:** «Опубликовать v1» / «Зафиксировать v2», панель «N предложений, M комментариев», diff по клику.
