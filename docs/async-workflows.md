# Async-подрежимы

Внутри `collaborationMode: async` — три подрежима. Выбираются при создании документа.

**По умолчанию — [round](#round-по-раундам).** [Handoff](#handoff-расширение) и [owner hub](#owner-hub-расширение) — опциональные расширения поверх round ([ADR-015](./decisions.md#adr-015-round--базовый-async-подрежим), [ADR-013](./decisions.md#adr-013-handoff-и-owner-hub-как-расширения-round)).

Общая основа — [ADR-011](./decisions.md#adr-011-head-draft-и-submit), [ADR-012](./decisions.md#adr-012-редактор-и-просмотр-правок-async-mvp-на-md).

## Общая модель

| Понятие | Смысл |
|---------|--------|
| **head** | Принятая версия (vN), immutable |
| **draft** | Рабочая копия текущего раунда; автосейв не меняет head |
| **Раунд** | От начала правок до `fixVersion` |
| **fixVersion** | Конец раунда: снимок draft → vN+1, документ снова открыт |
| **Effective content** | Draft, если есть незафиксированные изменения; иначе head (чтение, LLM) |

## Round (по раундам)

> **Статус:** базовый подрежим, по умолчанию ([ADR-015](./decisions.md#adr-015-round--базовый-async-подрежим)).

**Метафора:** документ с чекпоинтами — один пишет, фиксирует, любой может начать следующий раунд.

### Поток

1. Создание → **v1**, draft = содержимое v1, `activeEditorId = null`.
2. Участник (человек или LLM) начинает правки → `activeEditorId` = этот актор; правки в **shared draft**.
3. Другие участники **не пишут**, пока активен чужой lock (пользователь может перехватить lock приоритетно).
4. **`fixVersion`** → новая версия, `activeEditorId = null`, draft = head.
5. Следующий раунд может начать **любой** участник, в том числе тот же.

Слой `Edit` **не используется**. Комментарии к версиям — по [ADR-004](./decisions.md#adr-004-комментарии-привязаны-к-версии).

### Когда выбирать

- Документ как **память для LLM** (агент правит → пользователь корректирует когда нужно).
- Один файл контекста, история по версиям.
- Соавторы без жёсткой очереди «мяч у тебя».
- Любой сценарий, где достаточно «один редактор за раз + фиксация».

### LLM

- **Read:** effective content (draft или head).
- **Write:** только при `activeEditorId === llm` (или при захвате lock в начале агентского раунда).
- **История:** отдельные tools — `list_versions`, `get_version`; не обязательна в каждом вызове.
- Конец агентского раунда → `fixVersion` (summary опционально; позже — автогенерация LLM).

### UX (ориентиры)

- Между раундами: «документ готов к правкам».
- Во время раунда: кто редактирует (`activeEditorId`).
- Autosave черновика — без отдельной кнопки в UI.
- **«Сохранить»** (`fixVersion`) — единственная завершающая кнопка; создаёт vN+1, summary опционально.
- **«Редактировать»** — захват lock (round, командный режим); в solo — не показывается.
- **«Выйти без сохранения»** — `releaseEditLock`, редко, в меню карандаша.

### Термины в UI (round)

| UI | API / модель |
|----|----------------|
| Редактировать | `acquireEditLock` |
| Сохранить | `fixVersion` |
| Выйти без сохранения | `releaseEditLock` |
| (тихо) синхронизация черновика | `updateDraft` / autosave |

## Handoff (расширение)

> **Статус:** опционально при создании; целевой UX пересылки Word ([ADR-013](./decisions.md#adr-013-handoff-и-owner-hub-как-расширения-round)).

**Метафора:** пересылка Word — «мяч у тебя», явная очередь.

Наследует **round** (эксклюзивная запись, `fixVersion` = конец раунда) и **добавляет**:

- `currentActorId` — у кого ход по очереди;
- личный **fork** от head для актора с ходом (вместо одного shared draft);
- «Зафиксировать и **передать**» → vN+1 + смена `currentActorId` + `handoff.to`;
- reclaim, confirm при передаче без submit ([ADR-011](./decisions.md#adr-011-head-draft-и-submit)).

### Поток

1. Создатель → **v1**, `currentActorId` = создатель.
2. Актор с ходом правит **свой** fork, комментирует версии.
3. **fixVersion** + передача → новая версия + следующий `currentActorId`.
4. Следующий участник видит diff, правит, фиксирует и передаёт.

### Когда выбирать

- Два равных соавтора, быстрый пинг-понг.
- Юрист/редактор как **соавтор**, а не советник.
- Нужна дисциплина «ход у конкретного человека».

### Крайние случаи

- Получатель не отвечает → создатель может **забрать ход** (reclaim).
- Передача / reclaim при несохранённом submit → confirm.
- Цепочка B → C → A — последовательная передача.

### UX (ориентиры)

- Badge «Твой ход».
- Кнопка «Зафиксировать и передать» (кому + summary).
- Confirm при передаче без submit.

## Owner hub (расширение)

> **Статус:** опционально при создании; арбитраж при 3+ и конфликтах ([ADR-013](./decisions.md#adr-013-handoff-и-owner-hub-как-расширения-round)).

**Метафора:** maintainer — при несовместимых правках **кто-то должен принять решение**.

Наследует **round** (head, версии, фиксация) и **добавляет**:

- личные fork'и участников (`actorDrafts[]`);
- слой **Edit** (submit → proposal → apply/reject);
- только **owner** правит канонический draft и вызывает `fixVersion`;
- видимость fork'ов до submit — по [ADR-014](./decisions.md#adr-014-видимость-draft-и-публикация).

### Поток

1. Owner создаёт документ → **v1**.
2. Участники **параллельно**: комментарии + **submit** (`Edit`) из своего fork.
3. Owner видит submitted, diff `head ↔ каждый Edit`, собирает свой draft.
4. Совместимые правки — merge; конфликт — apply/reject вручную.
5. Owner **fixVersion** → v2. Edit к v1 → `superseded`.

### Когда выбирать

- Один ответственный + много ревьюеров.
- Product brief, политика, договор (owner подписывает итог).
- Несколько LLM-вариантов с одной базы.
- Параллельная обратная связь без очереди.

### Слияние в одну ветку

Паттерн «open branches» сворачивается в owner hub: sibling-версии → proposals вне main; owner фиксирует v2 с одним parent.

## Сравнение подрежимов

| | Round | Handoff | Owner hub |
|---|-------|---------|-----------|
| По умолчанию | **да** | опция | опция |
| Draft | Один shared | Fork на актора с ходом | Fork per участник + draft owner |
| Кто пишет | Один (`activeEditorId`) | Один (`currentActorId`) | Owner в канон; остальные — proposals |
| Следующий раунд | Любой участник | Конкретный по очереди | Owner после merge |
| Параллельные правки | Нет | Нет | Да (proposals) |
| Слой Edit | Нет | Нет | Да |
| LLM | Память, один агент | Агент по очереди | Несколько proposals |
| Сложность UI | Низкая | Низкая | Средняя (панель входящих) |

## Примеры

### Память LLM + правки пользователя

- **Round** — агент правит в раунде, фиксирует; пользователь правит когда нужно; история в versions.

### Договор: вы + юрист

- **Owner hub**, если вы ответственный за текст; юрист шлёт proposal.
- **Handoff**, если юрист — полноценный соавтор по очереди.

### Бриф: вы + 5 коллег

- **Owner hub** — параллельные proposals, вы собираете v2.

### Пост: два маркетолога

- **Handoff** — пинг-понг с явной передачей хода.
- **Round** — если очередь не нужна, достаточно «кто успел взять lock».

### Текст + несколько LLM

- **Owner hub** — варианты как proposals, diff с v1.

### Политика: HR + юрист + security

- **Owner hub** — HR owner, остальные proposals + комментарии.
